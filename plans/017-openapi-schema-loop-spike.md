## Why this matters

The root README promises an "OpenAPI schema export command and schema
artifact workflow for client generation". The generated project's workflow
exports `openapi-internal.json` and `openapi-v1.json` on every PR/push —
and stops at `actions/upload-artifact`. Consumers get a schema buried in
per-run CI artifacts: no reviewable schema diffs on PRs, no drift signal
when an endpoint changes contract, no generated client. The promise is one
step short of delivered. This spike picks the lightest step that delivers
it.

## Steps

### Step 1: Establish determinism

Bake a default project, run the two export commands twice in a row (mock env
exactly as the workflow does — no Postgres needed per the workflow's own
env), and diff. If output is not byte-stable across runs, every
diff-based option needs a canonicalization step — measure it, don't assume.

### Step 2: Evaluate the options

- **A. Commit the schemas + drift check** (recommended candidate): schemas
  live at e.g. `docs/openapi/openapi-v1.json` in the generated repo; the
  export workflow re-exports and `git diff --exit-code`s them, failing when
  a PR changes the contract without updating the committed schema. PR
  reviewers see contract changes as diffs. No credentials, no push-from-CI.
  Costs: bake-time bootstrap (post-gen hook or first-commit step), and the
  schemas must be regenerated by developers (document the command).
- **B. Client-gen knob**: e.g. `use_typescript_client=yes` generating a
  typed client in CI. Honest cost: heavy (new knob, toolchain pin, artifact
  publishing story) — likely a "later, if users ask".
- **C. Release asset**: attach schemas to the release workflow. Cheap but
  only helps at release granularity; PR-time drift stays invisible.

For each: files touched, whether the generated workflow stays Jinja-free,
interaction with the 100%-coverage gate and pre-commit, effort (S/M/L).

### Step 3: Write the design doc

`plans/017-openapi-loop-design.md`: determinism evidence, options table,
recommendation with rationale, implementation sketch for the recommended
option (exact workflow steps, bootstrap mechanism, developer runbook line
for the generated README), and open questions.
