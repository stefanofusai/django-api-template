# Implementation Plans

Generated by the improve skill on 2026-07-08 from a deep audit planned at
commit `eee3978`. This directory contains the active plan cycle only.

Plans are numbered in recommended execution order, restarted from 001 for this
cycle. Priority bands come first, with dependency-consistent ordering within
each band:

- 001-002: P1 fixes.
- 003-005: P2 hardening and local guardrails.
- 006-008: P3 design, verification, and documentation follow-ups.

Plan 008 was added from the audit's qualitative feedback after the first seven
plans and documents the `_anon_budget_exhausted` coupling to ninja-extra's
private cache format.

Plans 010-019 were added on 2026-07-08 (evening) by a second deep audit at
commit `75c4dce`. That audit's headline: the GENERATED project is clean
(correctness, security, performance, and dependency sweeps returned nothing
new, with two never-baked knob combos empirically verified and a clean
pip-audit), but the template's OWN machinery — hooks, guard scripts, boot
guards, CI matrices — has systematically weaker verification than the
projects it generates. 010-014 close that gap; 015 is the one direction item
promoted to a build plan; 016-019 are design spikes.

**Context every executor needs**: this repo is a **cookiecutter template**.
Files under `{{cookiecutter.project_slug}}/` contain Jinja and cannot be run
directly - verification means baking a project
(`uvx cookiecutter . -o /tmp/verify --no-input <knobs>`) and running its suite
there. Always quote paths containing `{{cookiecutter.project_slug}}` in shell
commands. `.github/workflows/*` and `.agents/*` inside the project directory
are `_copy_without_render` - pure YAML/Markdown, no Jinja allowed there.

## Findings considered and rejected

Recorded so nobody re-audits them:

- **`api_auth` knob is a byte-identical no-op when `use_example_api=no`**:
  by-design and documented in the root README's Variables table.
- **CSP `unsafe-inline` as a vulnerability**: downgraded - the README/knob
  prompt explicitly document the starter policy as allowing inline
  scripts/styles; the improvement path is plan 007's nonce spike, not a
  security fix.
- **Root AGENTS.md naming example says `Template CI` while the workflow is
  named `CI`**: one-word doc fix, fold into any future docs pass.
- **`rtk` references in the generated AGENTS.md**: initially left to
  maintainer preference; maintainer decided on 2026-07-08 to remove them —
  now item F of plan 004 (root AGENTS.md keeps its rtk guidance).
- **Smoke job rebuilds the prod image cold instead of reusing the build job's
  gha cache**: real redundancy, but wiring BuildKit `cache_from` into compose
  build is fiddly for a few saved CI minutes; jobs run in parallel so
  wall-clock is unaffected. Not worth doing now.
- **`traefik_tls=external` is never boot-tested** (bake+lint only): needs real
  certs; accepted gap, revisit only if a self-signed-cert smoke variant
  becomes cheap.

Rejected by the second (75c4dce) audit on 2026-07-08:

- **Compose healthcheck/service dedup** (byte-identical Postgres/Redis/api
  healthcheck blocks across dev/prod/ci-services): real duplication, but the
  files legitimately diverge, Jinja templates cannot share YAML anchors
  across files, and a Compose `include:` base layer costs more risk than the
  drift it prevents. Revisit only if a healthcheck-drift bug actually bites.
- **Root `.editorconfig`**: pre-commit hooks already enforce
  whitespace/EOL/format at commit time; marginal editor-side value.
- **Notes list endpoint returns full `body` per row**: defensible for an
  example API whose purpose is demonstrating the vertical slice; a
  list/detail schema split would complicate the exemplar. By-design.
- **axes.W001 in CI bakes** (django-axes flags the CI-only locmem cache):
  inert — prod uses Redis; the CI overlay is not a lockout surface.
