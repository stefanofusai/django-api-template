## Recommended order & independence

All four plans are **independent** — no hard dependencies between them. Suggested
order by "land immediately vs. needs a decision":

1. **001 (gitleaks)** — smallest, self-contained, pure win. Do first.
2. **002 (behind_proxy)** — the one genuine security nuance; contains a
   maintainer *design decision* (should a plain-HTTP prod mode exist?) flagged
   as a STOP. Resolve that before implementing.
3. **004 (auth spike)** — highest *strategic* value for an API template, but it
   is a **design spike**: it produces a proposal + throwaway PoC and STOPs for
   maintainer decisions. Its output feeds a future build plan (`005-…`). Start
   it early if the auth direction matters; it does not block 001–003.
4. **003 (OpenAPI export)** — nice-to-have DX/deliverable; lowest urgency.

## Findings considered and rejected (so they aren't re-audited)

- **No CORS / throttling / default auth**: deliberate, documented design
  decisions (README "Design Decisions"). Correct to omit — not defects. (Token
  auth as an *opt-in example* is a separate, forward-looking item → plan 004.)
- **`DEFAULT_AUTO_FIELD` not set anywhere**: not a bug. The `core` migration
  freezes `BigAutoField` for `User.id` and the migration-drift CI gate is green,
  proving Django 6.0 resolves the default to `BigAutoField` (no `W042`). Clean
  omission.
- **`update_note` uses full `save()` not `save(update_fields=…)`**
  (`apps/notes/routes.py`): negligible on a 3-field model; not worth a plan.
- **`/ready` does not check the Celery broker**: correct by design — the web
  readiness probe should gate on what the request path needs (DB + cache), not
  the worker's broker (a separate service).
- **`deploy-check.sh` runs `check --deploy --tag=security` only**: standard
  recommendation; the full `--deploy` set adds W-series noise. Fine as-is.
- **`.env.example` ships `localhost`/`127.0.0.1` in `ALLOWED_HOSTS` and a
  `django-insecure-` `SECRET_KEY`**: well-handled — prod boot-guard rejects the
  insecure prefix, and the file is documented as replace-before-deploy.
