# Plan 021 Design: Vendored Skill Hash Verification

## Status

Recommendation: do not use `npx skills verify`; it is not available in the
current `skills` CLI. Add a repo-owned tree-hash verifier instead, and wire it
into both repo-root pre-commit and CI if the maintainer accepts the gate.

The spike drift check was clean:

```text
$ git diff --stat ae42991..HEAD -- "{{cookiecutter.project_slug}}/skills-lock.json" .github/workflows/ci.yaml .pre-commit-config.yaml
<no output>
```

Note on command execution: plain `rtk npx ...` routed through rtk's npm wrapper
and did not invoke the real package command. The captured CLI checks below used
`rtk proxy npx ...`, which preserves the required `rtk` prefix while running the
actual `npx` command.

git diff in this worktree, but it is another reason not to use the current CLI as
a verification gate.

## Recommended Mechanism

Add a small self-hash verifier controlled by this repository.

Use `skills-lock.json` for the expected value by adding a new field per skill,
for example `localTreeHash`, while leaving the CLI-owned `computedHash` intact:

```json
{
  "version": 1,
  "skills": {
    "django-expert": {
      "source": "vintasoftware/django-ai-plugins",
      "sourceType": "github",
      "skillPath": "plugins/django-expert/skills/SKILL.md",
      "computedHash": "a6b8c224017570496d688a526fcbea74025a01cc90d9afb0977bfcc1f76af306",
      "localTreeHash": "8188ebe2e294f7186e132d1aaa67a20fa5748fc4ed2a12929e7f68536dcfc245"
    }
  }
}
```

Hash algorithm:

1. For each skill name in `skills-lock.json`, read tracked files under
   `.agents/skills/<name>/`.
2. Sort paths by their relative path inside that skill tree.
3. Hash each file's bytes with SHA-256.
4. Hash a NUL-separated stream of `path:<relative-path>` and
   `sha256:<file-hash>` records.
5. Compare that tree hash to `localTreeHash`.

This checks the actual vendored tree, including reference files, without relying
on the upstream tool's `computedHash` semantics.

If the maintainer prefers not to extend the `skills` CLI's lock schema, use an
adjacent `skills-integrity.json` instead. I prefer `localTreeHash` in
`skills-lock.json` because it keeps source metadata and local integrity metadata
on the same skill record, and any future CLI rewrite that drops the field will
fail the gate until the maintainer intentionally re-baselines.

## Proposed Gate

Add a helper script in the rendered project, for example
`.github/scripts/verify-skill-tree-hashes.py`, and use the same script in the
template root by passing the literal template project directory as an argument.
The script can be pure Python stdlib: `json`, `hashlib`, `pathlib`, and
`subprocess` for `git ls-files`.

Repo-root pre-commit hook:

```yaml
  - repo: local
    hooks:
      - id: verify-skill-tree-hashes
        name: verify vendored skill tree hashes
        entry: python .github/scripts/verify-skill-tree-hashes.py {{cookiecutter.project_slug}}
        language: python
        pass_filenames: false
        always_run: true
```

Repo-root CI step:

```yaml
      - name: Verify vendored skill tree hashes
        run: python .github/scripts/verify-skill-tree-hashes.py '{{cookiecutter.project_slug}}'
```

Generated-project pre-commit hook:

```yaml
  - repo: local
    hooks:
      - id: verify-skill-tree-hashes
        name: verify vendored skill tree hashes
        entry: python .github/scripts/verify-skill-tree-hashes.py .
        language: python
        pass_filenames: false
        always_run: true
```

Generated-project CI does not need a separate explicit step if the generated
pre-commit stack runs in CI, which it already does. That gives downstream users a
local and CI guard without adding another workflow dependency.

## Scope Decision

Recommended scope: enforce both the template repo's vendored skills and baked
projects.

Reasoning:

- The template repo is where accidental edits and re-vendor drift are most
  likely to enter.
- Baked projects carry `.agents/skills/` and `skills-lock.json`, so downstream
  drift can happen after generation too.
- A generated pre-commit hook is enough for baked-project CI because the
  generated workflow already runs `uv run pre-commit run --all-files`.
- The check is standalone and does not require un-excluding `.agents/` from Ty,
  Ruff, markdownlint, or other broad linters.

## Maintenance Notes

Plans 003 and 012 intentionally change the vendored skill set in different ways.
Any accepted hash gate must be re-baselined after those plans land:

- Plan 003 prunes `django-celery-expert` from some bakes.
- Plan 012 adds three more vendored skills.

For future re-vendoring, the runbook should be:

1. Re-vendor with `npx skills experimental_install` or the accepted replacement.
2. Review the `.agents/skills/` diff.
3. Recompute `localTreeHash` values.
4. Run the verifier clean.

Dependabot or Renovate support for skills.sh sources would complement this gate,
but it should be future work. The gate here only detects local drift against the
committed baseline.

## Open Questions For Maintainer

1. Should the repo extend `skills-lock.json` with `localTreeHash`, or keep the
   repo-owned hashes in a separate `skills-integrity.json` file?
2. Should generated projects carry the verifier by default, or should this guard
   be template-repo-only?
3. Should the accepted implementation be Python for portability with the existing
   Python template stack, or shell for closer parity with the scratch prototype?
4. Should `npx skills experimental_install --help` mutating the tree be reported
   upstream before relying on any more CLI behavior?
5. After plans 003 and 012 land, should this gate be implemented once against
   the final intended skill set rather than in between those changes?
